# Android XML vs Jetpack Compose

## XML

### 장점
- **성숙한 기술**: 오랜 기간 사용되어 검증된 기술로, 안정성이 높다.
- **풍부한 도구 지원**: 다양한 레이아웃 에디터와 디자인 도구를 활용할 수 있다.
- **넓은 커뮤니티**: 풍부한 레퍼런스와 라이브러리를 통해 문제 해결이 용이하다.
- **높은 호환성**: 오래된 안드로이드 버전도 지원하여 폭넓은 사용자층을 확보할 수 있다.

### 단점
- **복잡한 UI 구현**: 프로젝트 크기가 커지고 복잡한 UI를 구현하려면 많은 XML 코드를 작성해야 하며, 코드가 장황해져 유지보수가 어렵다.
- **제한적인 유연성**: 동적인 UI 변경이나 복잡한 애니메이션 처리에 제약이 있다. (여러 .xml 파일을 생성하고 적용하면서 복잡해진다.)
- **성능 문제**: 뷰 계층 구조가 복잡해지면 성능 저하가 쉽게 발생할 수 있다. 코드가 장황해지면서 레이아웃 구조가 효율적이지 않을 확률이 높아진다.

<br><br>

## Jetpack Compose

### 장점
- **높은 개발 생산성**: 간결한 코드로 빠르게 UI를 개발하고 유지보수할 수 있습니다. (예 : RecyclerView와 LazyColumn 코드 양 비교)
- **뛰어난 유연성**: 동적이고 복잡한 UI를 쉽게 구현하고 애니메이션 처리도 용이합니다. (모든 것을 Kotlin 코드로 구현하며 직관적이다.)
- **높은 성능**: 컴파일러 최적화를 통해 효율적인 렌더링과 빠른 성능을 제공합니다.
- **쉬운 테스팅**: 컴포저블 단위로 UI 테스트가 용이하며, 유지보수가 쉽습니다.

### 단점
- **높은 학습 곡선**: 새로운 패러다임에 대한 이해가 필요하며, 초기 학습 비용이 높습니다.
- **제한적인 호환성**: 안드로이드 5.0 (API 21) 이상에서만 사용할 수 있습니다.
- **상대적으로 작은 커뮤니티**: XML에 비해 커뮤니티 규모가 작고 레퍼런스가 부족할 수 있습니다. (점점 개선되고 있는 단점)

<br><br>

## 비교 요약

| 특징    | XML                              | Compose                        |
|-------|----------------------------------|--------------------------------|
| 선언 방식 | XML MarkUp 언어 기반 명령형 UI          | Kotlin 코드 기반 선언형 UI            |
| 생산성   | 복잡한 UI 구현 시 코드량 급격히 증가, 유지보수 어려움 | 비교적 간결한 코드, 빠른 UI 개발, 쉬운 유지보수  |
| 유연성   | 제한적인 동적 UI 변경                    | 동적이고 복잡한 UI 구현 용이, 애니메이션 처리 용이 |
| 성능    | 계층 구조 복잡성에 따른 성능 저하 가능성 높음       | 컴파일러 최적화, 효율적인 렌더링, 빠른 성능      |
| 학습 곡선 | 익숙하고 비교적 쉬운 학습                   | 새로운 패러다임, 초기 학습 비용 존재          |
| 도구 지원 | 다양한 레이아웃 에디터, 디자인 도구             | 실시간 미리보기, 코드 기반 디자인 도구         |
| 테스팅   | UI 테스트 복잡                        | UI 테스트 용이                      |

<br><br>

## 결론
XML은 안정적이고 호환성이 높은 장점이 있지만, 복잡한 UI 구현과 유지보수에 어려움이 있다. 
Compose는 높은 개발 생산성과 유연성을 제공하지만, 학습 곡선이 높고 호환성이 제한적이다.
